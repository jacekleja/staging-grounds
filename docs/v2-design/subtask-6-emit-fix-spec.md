# Subtask 6 — Emit Fix-Spec: gift_advisor / comparison turn-1 takeover blocks (C-04..C-07)

ONE-SENTENCE ROOT CAUSE: The gift/advice/comparison turn-1 handlers stream the LLM opener via `llm.astream` (partials) and then append the deterministic `lbjson` block ONLY into the node's RETURNED `final_message` (`graph.py:1704`), which can reach the wire only via the LangGraph `messages/complete` event — and that complete event does NOT deliver the appended block to the FE (CONFIRMED by live symptom + the proxy's A1-guard / messages-complete handling at `app/clients/langgraph_client.py:415-436`). This is a STREAM-DELIVERY bug, NOT a routing bug and NOT a zero-results bug. (The EXACT drop step is NOT byte-confirmed — see "Root cause" fact 3 and HK2: the originally-suspected `complete_msg_id in seen_partial_msg_ids` disjunct is RULED OUT because the returned `AIMessage` at `graph.py:1704` is constructed with no `id=` and therefore carries a FRESH id, not the streamed opener's id.)

## Root cause
Confirmed mechanism, three load-bearing facts:

1. **Routing is CORRECT.** `mode_dispatch` (`graph.py:2929`) writes `mode=gift_advisor` to state (`:2940`); `dispatch_route` (`:2948`) sends every non-deflect mode to `regular_turn` = `handle_regular_turn`; inside it, `:2196` `if state.get("mode")==_GIFT_ADVISOR_MODE and state.get("is_first_turn",False)` calls `_handle_gift_advisor_turn1` (`:1665`). The live `prompt_fingerprint="local-gift-advisor-turn1-prompt@...+mode-gift_advisor"` (computed at `:1714` with `prompt_identity="gift-advisor-turn1-prompt"`) PROVES `_handle_gift_advisor_turn1` was entered. The branch fires. Comparison routes the same way via `:2220` → `_handle_comparison_turn` (`:1876`).

2. **The block IS built and returned, but only in `final_message`.** `_handle_gift_advisor_turn1` streams the LLM opener via `llm.astream` (`:1694`), then constructs `final_message = AIMessage(content=f"{opener}\n```lbjson\n{takeover_json}\n```")` (`:1704`) and returns it via `"messages": Overwrite(all_messages + [final_message])` (`:1748-1749`). The deterministic lbjson suffix never passes through `astream` — it exists only on the returned/complete message. `_handle_advice_turn1` (`:1766`) and `_handle_comparison_turn` (`:1876`/`:1889`) use the IDENTICAL pattern.

3. **The complete-event delivery of the appended block fails on the wire (CONFIRMED), but the EXACT drop step is INFERRED-NOT-BYTE-CONFIRMED.** `app/clients/langgraph_client.py` streams `stream_mode=["messages","custom"]` (`:265`). The A1 guard (`:415-420`): on `messages/complete`, if `complete_msg_id is None or complete_msg_id in seen_partial_msg_ids: continue`. **Re-derivation against code rules OUT the `seen_partial_msg_ids` disjunct:** the returned `final_message = AIMessage(content=...)` at `graph.py:1704` is constructed with NO `id=` argument, so LangChain assigns it a FRESH UUID — distinct from the streamed-opener `AIMessageChunk` id (`llm.astream`, `graph.py:1694`) that populated `seen_partial_msg_ids`. A fresh id is neither `None` nor in `seen_partial_msg_ids`, so `:419` should NOT fire on it. Candidate drop steps that remain LIVE (one of these, not byte-confirmed which): (a) `complete_msg_id is None` on the wire (the Overwrite-returned message arrives id-stripped through LangGraph's messages/complete serialization); (b) the complete event survives `:419` but its full cumulative content re-enters the hydrator at a `_raw_text_length`/msg-id-boundary mismatch (`:466-477`, `:532-545`) and the delta computed against the new fresh msg_id surfaces only the opener, never re-feeding the fence; (c) the messages/complete for an `Overwrite` state update is not emitted as a user-visible AI-text event at all. CONFIRMED regardless of which: the live SSE shows the streamed opener tokens and NO `lbjson` fence — the appended block does not reach the FE. (HK1/HK2 track this open discrimination; the recommended fix below is robust to ALL of (a)/(b)/(c) because it does not depend on the messages/complete path.)

This explains the contradicting datum the delegation flagged: the prior "composition=question_led is by-design; FE keys off lbjson shape" hypothesis is IRRELEVANT to why the block is absent — the block never reaches the FE to be keyed off at all. The composition field is a red herring for C-04..C-07.

## The fix-spec (minimal change set)

The block is constructed correctly and the routing is correct. The ONLY defect is that the deterministic lbjson suffix is appended to the node's RETURNED message and reaches the wire only via `messages/complete`, which fails to deliver it (see Root cause fact 3). Two mechanically distinct approaches close C-04..C-07.

**PEER-REVIEW CORRECTION (supersedes the prior agent-only Approach A):** An agent-only custom-event emit of a NEW kind is NOT implementable as written. The proxy custom-event dispatch (`app/clients/langgraph_client.py:279-376`) handles ONLY the known kinds (`lbx.search_context`, `lbx.work_status`, `lbx.no_preview`, `lbx.deflection_text`, `lbx.turn_classification`); the terminal branch at `:375-376` is `# Unknown kind or version: silently skip. Custom events never yield. / continue`. A new `lbx.lbjson_block` would hit that branch and be DROPPED — never forwarded, never hydrated. Furthermore the ONE custom kind that yields text, `lbx.deflection_text` (`:321-336`), does `self.full_response_text += content; yield content` but does NOT call `self._hydrator.process_token(...)` — so even a text-yielding custom event bypasses `LbjsonHydrator`, which runs only on the `messages/partial|complete` text path (`:537`). **Therefore any correct fix MUST touch the proxy repo.**

**⚠ THIS FIX TOUCHES THE PROXY REPO (`conversational-proxy`).** Both viable approaches below change proxy code under `conversational-proxy/app/clients/langgraph_client.py`. **C-21 requires a SEPARATE proxy commit** — the implementer and the housekeeping commit step MUST stage and commit the proxy-repo changes independently from the agent-repo changes. Amended Approach A additionally touches the agent repo (custom_events.py + three graph.py handlers); Approach B is proxy-repo-only.

**Multiple viable approaches:**

- **Amended Approach A (RECOMMENDED — new custom kind, emitted agent-side AND handled proxy-side through the hydrator).** Closes the gap WITHOUT depending on the (unconfirmed) messages/complete drop step, so it is robust to all of Root-cause-fact-3 (a)/(b)/(c).
  - AGENT repo: add `_emit_lbjson_block(block_text: str, source_node: str)` in `custom_events.py` (model on the existing `_emit_search_context`/`_emit_turn_classification` via `get_stream_writer()`), writing `{"kind": "lbx.lbjson_block", "version": 1, "content": block_text, "source_node": source_node}`. Call it in all THREE handlers immediately after the fence string is built and BEFORE returning `final_message`: `_handle_gift_advisor_turn1` (`graph.py` ~:1703, with the `\n\`\`\`lbjson\n{takeover_json}\n\`\`\`` fence — emit the SAME fenced string, not bare JSON, so the hydrator's fence parser triggers), `_handle_comparison_turn` (~:1889, side_by_side fence), `_handle_advice_turn1` (~:1813-1814, advice takeover fence). Keep `final_message` as-is for checkpoint fidelity.
  - PROXY repo: in `app/clients/langgraph_client.py`, add a new `elif` branch in the custom-event dispatch BEFORE the `:375-376` unknown-kind skip (insert at ~:337, alongside the other kind handlers), matching `payload.get("kind") == "lbx.lbjson_block" and payload.get("version") == 1`. In that branch, feed the content through the SAME hydrator/full_response_text/yield path message text uses: `content = payload.get("content"); hydrated = self._hydrator.process_token(content, self._cached_search_results); if hydrated is not None: self.full_response_text += hydrated; yield hydrated`. (Feeding via `process_token` — not the bare `+= content; yield content` deflection pattern — is REQUIRED so the lbjson fence is hydrated against `_cached_search_results`, identical to inline message-text fences.) Because `_emit_search_context(products=[], ...)` already fires in these handlers (`graph.py:1691`), `_cached_search_results` is correctly empty for the gift/comparison anchored-chip case.
  - Touches: BOTH repos — `custom_events.py` + three `graph.py` handlers (agent) AND `langgraph_client.py` (proxy, separate C-21 commit).

- **Approach B (proxy-repo-only — forward the un-streamed suffix in the messages/complete path).** In `app/clients/langgraph_client.py:415-436`, when a `messages/complete` AI event arrives whose content extends beyond what was already streamed for its msg_id, compute the un-streamed suffix and feed ONLY that suffix through `self._hydrator.process_token(...)` + `yield`, instead of the current early `continue`. CAVEAT — this is NOT a simple "narrow the `seen_partial_msg_ids` disjunct" change as the prior draft claimed: per Root-cause fact 3, `final_message`'s fresh id is NOT in `seen_partial_msg_ids`, so that disjunct does not even fire here; the actual drop step is unconfirmed (id-None on the wire / msg-id-boundary delta mismatch / complete-not-emitted). Approach B must FIRST instrument and confirm WHICH step drops the block (HK1/HK2), THEN target it. Higher regression risk: touches the shared streaming path for ALL message types and providers; the partial/complete cross-msg-id diffing (`:466-505`, `:532-545`) is fiddly and the hydrator is stateful mid-stream. Dispreferred until the exact drop step is byte-confirmed.

**Recommendation: Amended Approach A.** Lower total risk despite touching two repos: it is robust to ALL candidate drop steps (does not depend on diagnosing the messages/complete failure), it reuses the proven custom-event + hydrator mechanisms, and its proxy change is ADDITIVE (a new `elif` branch that cannot regress existing kinds) rather than a mutation of the shared complete-message path that Approach B requires.

**Architect escalation NOT required.** The abstraction is sound (deterministic-block-append, custom-event streaming, and the LbjsonHydrator all already exist); this is a missing wire-up of existing mechanisms across the agent/proxy boundary, squarely in implementer scope. Scope is local: three agent handlers + one new helper in `custom_events.py` + one new additive `elif` branch in the proxy's `langgraph_client.py` custom-event dispatch.
